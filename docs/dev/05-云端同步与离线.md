# 云端同步与离线缓存

> 本文档描述 iPaper 的多端同步和离线缓存架构。第一波（认证 + 数据隔离 + 偏好上云）已完成，本文档记录**第二波待实施**的内容。

## 当前架构

```
Electron 桌面端                     Web / iPhone
  本地后端 (:3000)                    │
  本地数据 (~/.ipaper/data/<uid>/)    │
  免登录（本地模式）                   │
                                     ▼
                              VPS (www.moshang.xyz)
                              Nginx → FastAPI (:3000)
                              数据: ~/.ipaper/data/<uid>/
                              JWT 认证 + 邀请码注册
```

- Electron 端和 Web 端目前是**独立的**，数据不互通
- Web 端无离线能力，断网后无法查看任何内容

## 第二波目标

1. **Web 端 IndexedDB 离线缓存**（优先级：高）
2. **Electron 双向同步 API**（优先级：中）

两者可并行开发，前端改 `frontend/`，后端改 `backend/`，文件不冲突。

## 任务 1：Web 端 IndexedDB 离线缓存

### 目标

Web/iPhone 端可离线查看已访问过的论文 PDF 和对话历史。

### 新增文件

**`frontend/src/services/cache.ts`** — IndexedDB 封装：
- 使用 `idb` 库（轻量 IndexedDB wrapper，需 `npm install idb`）
- 数据库 schema：
  - `papers` store：论文元信息（PaperListItem）
  - `pdfs` store：PDF Blob（key = `paperId:lang`）
  - `chats` store：对话历史 JSON
  - `preferences` store：偏好设置

**`frontend/src/services/offlineApi.ts`** — 离线 API 代理层：
- 包装 `api.ts` 的请求函数
- 联网时：正常请求，成功后将响应写入 IndexedDB
- 断网时：从 IndexedDB 读取缓存返回
- 写操作断网时：排入队列（存 IndexedDB），联网后重放

**`frontend/public/sw.js`** — Service Worker：
- 缓存前端静态资源（HTML/JS/CSS/字体）
- 使页面本身可离线加载
- 注意：不缓存 API 请求（由 offlineApi 层处理）

### 修改文件

- `frontend/src/services/api.ts` — 导出的函数包一层离线代理
- `frontend/src/main.tsx` — 注册 Service Worker
- `frontend/package.json` — 新增 `idb` 依赖

### 缓存策略

- PDF：用户打开过的论文 PDF 自动缓存（authFetch 下载为 blob 后同时写 IndexedDB）
- 对话历史：每次 loadHistory 时缓存
- 偏好：登录后立即全量缓存
- 容量管理：超过 500MB 时按 LRU 清理最旧的 PDF
- iPhone Safari 大约允许每站点 1GB

## 任务 2：Electron 双向同步 API

### 目标

Electron 本地后端与云端后端之间同步数据，使本地新增的论文/对话能出现在 Web 端，反之亦然。

### 同步策略

- **时间戳驱动**：每条数据带 `updated_at` 字段
- **冲突解决**：Last-Write-Wins（以 `updated_at` 更大者为准）
- **同步粒度**：论文级别（一篇论文的元信息 + PDF + 所有会话作为一个同步单元）

### 新增后端文件

**`backend/routers/sync.py`** — 同步 API 路由：
- `GET /api/sync/manifest` — 返回该用户所有论文的 `{arxiv_id, updated_at}` 清单 + 偏好和画像的 `updated_at`
- `GET /api/sync/papers/<id>/bundle` — 下载某篇论文的完整数据包（meta + PDF + 所有聊天 JSON，zip 格式）
- `PUT /api/sync/papers/<id>/bundle` — 上传某篇论文的完整数据包
- `GET /api/sync/preferences` — 下载偏好
- `PUT /api/sync/preferences` — 上传偏好
- `GET /api/sync/profile` — 下载画像
- `PUT /api/sync/profile` — 上传画像

**`backend/services/sync_service.py`** — 同步逻辑封装：
- 生成 manifest（遍历用户论文目录，读取 updated_at）
- 打包/解包论文 bundle
- 比较时间戳判断同步方向

### 修改后端文件

- `backend/main.py` — 注册 sync 路由
- `backend/config.py` — 新增 `sync_url`（云端 API 地址）和 `sync_token`（JWT）配置项
- 现有数据文件需新增 `updated_at` 字段：`meta.json`, `sessions.json`, 各聊天 JSON

### Electron 端同步流程

```
启动时 → GET /sync/manifest → 对比本地 manifest
  ↓
云端有更新 → GET /sync/papers/{id}/bundle → 写入本地
本地有更新 → PUT /sync/papers/{id}/bundle → 推到云端
  ↓
同步 preferences 和 profile（同理对比 updated_at）
```

同步时机：
- 启动时全量对比
- 本地数据变更后异步推送（防抖）

## 注意事项

- 服务器 472MB 内存，sync bundle 传输大 PDF 时注意内存峰值
- `bcrypt` 版本锁定 4.2.1（和 passlib 1.7.4 兼容），见 `requirements.txt`
- 本地模式（host=127.0.0.1）免认证，sync API 同样需要处理本地模式的认证跳过
- 前端 `authFetch` 已处理 401 自动登出，离线代理层需要在 `authFetch` 之上再包一层
