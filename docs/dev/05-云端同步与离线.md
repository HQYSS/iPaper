# 云端同步与离线缓存

> 本文档描述 iPaper 的多端同步和离线缓存架构。第一波（认证 + 数据隔离 + 偏好上云）已完成，第二波（IndexedDB 离线缓存 + Sync API）**已完成基础实现**。

## 当前架构

```
Electron 桌面端                     Web / iPhone
  本地后端 (:3000)                    │
  本地数据 (~/.ipaper/data/<uid>/)    │
  免登录（本地模式）                   │
                                     ▼
                              VPS (www.moshang.xyz)
                              Nginx → FastAPI (:3000)
                              数据: ~/.ipaper/data/<uid>/
                              JWT 认证 + 邀请码注册
```

- Electron 端和 Web 端目前是**独立的**，数据不互通
- Web 端无离线能力，断网后无法查看任何内容

## 第二波目标（✅ 已完成基础实现）

1. **Web 端 IndexedDB 离线缓存** ✅
2. **Electron 双向同步 API** ✅

两者已并行完成，前端改 `frontend/`，后端改 `backend/`。

## 任务 1：Web 端 IndexedDB 离线缓存 ✅

### 目标

Web/iPhone 端可离线查看已访问过的论文 PDF 和对话历史。

### 已实现文件

**`frontend/src/services/cache.ts`** — IndexedDB 封装：
- 使用 `idb` 库（轻量 IndexedDB wrapper）
- 数据库 schema（6 个 store）：
  - `papers` store：论文元信息（PaperListItem），key = arxiv_id
  - `pdfs` store：PDF Blob（key = `paperId:lang`），含 size 字段用于 LRU
  - `chats` store：对话历史 JSON（单论文 key = `paperId:sessionId`，串讲 key = `cp:sessionId`）
  - `sessions` store：会话列表缓存
  - `preferences` store：偏好设置
  - `pendingOps` store：离线写操作队列，自增 key + createdAt 索引

**`frontend/src/services/offlineApi.ts`** — 离线 API 代理层：
- `networkFirstJson<T>()` — 通用 network-first + cache-fallback 策略
- `queuedWrite()` — 断网时写操作排入 pendingOps，联网后自动 replay
- `setupOfflineListeners()` — 监听 `online` 事件自动重放队列
- 为每个读取类 API 提供 `xxxOffline()` 对应函数
- `fetchPapersOffline()` 会在最终返回给 UI 前按 `download_time` 排序，保证论文库在在线/离线两条路径上都按添加时间显示，而不是受 IndexedDB `papers` store 的 `arxiv_id` 主键顺序影响

**`frontend/public/sw.js`** — Service Worker：
- 静态资源（JS/CSS/字体/图片）cache-first 策略
- HTML 导航请求 network-first + cache-fallback
- API 请求不拦截（由 offlineApi 层处理）
- 自动清理旧版本缓存

### 修改文件

- `frontend/src/services/api.ts` — 末尾 re-export 所有 offlineApi 函数
- `frontend/src/main.tsx` — 注册 Service Worker + setupOfflineListeners()
- `frontend/package.json` — 新增 `idb` 依赖

### 缓存策略

- PDF：用户打开过的论文 PDF 自动缓存（authFetch 下载为 blob 后同时写 IndexedDB）
- 对话历史：每次 loadHistory 时缓存
- 偏好：登录后立即全量缓存
- 容量管理：超过 500MB 时按 LRU 清理最旧的 PDF
- iPhone Safari 大约允许每站点 1GB

## 任务 2：Electron 双向同步 API ✅

### 目标

Electron 本地后端与云端后端之间同步数据，使本地新增的论文/对话能出现在 Web 端，反之亦然。

### 同步策略

- **时间戳驱动**：每条数据带 `updated_at` 字段（已在 meta.json、对话 JSON、sessions.json 中自动写入）
- **冲突解决**：Last-Write-Wins（以 `updated_at` 更大者为准）
- **同步粒度**：论文级别（一篇论文的元信息 + PDF + 所有会话作为一个同步单元）
- **删除传播**：使用 `deleted_papers.json` tombstone；删除也是一条带时间戳的同步事实，不再只是“本地把目录删掉”

### 已实现后端文件

**`backend/routers/sync.py`** — 同步 API 路由（接受本地免认证 / 云端 sync token / 云端普通 JWT）：
- `GET /api/sync/manifest` — 返回该用户所有论文的 `{arxiv_id, updated_at}` 清单、删除 tombstone 列表 `{arxiv_id, deleted_at}`，以及偏好和画像的 `updated_at`
- `GET /api/sync/papers/<id>/bundle` — 下载某篇论文的完整数据包（zip 格式，含 meta + PDF + 所有聊天 JSON）
- `PUT /api/sync/papers/<id>/bundle` — 上传某篇论文的完整数据包（上限 100MB）
- `DELETE /api/sync/papers/<id>` — 记录 tombstone 并删除论文目录
- `GET /api/sync/preferences` — 下载偏好
- `PUT /api/sync/preferences` — 上传偏好（自动添加 `updated_at`）
- `GET /api/sync/profile` — 下载画像
- `PUT /api/sync/profile` — 上传画像

**`backend/services/sync_service.py`** — 同步逻辑封装：
- `get_manifest()` — 遍历用户论文目录，读取 updated_at 生成清单
- `deleted_papers.json` — 删除 tombstone 存储；`get_manifest()` 会只暴露“仍然生效”的删除记录
- `create_paper_bundle()` / `extract_paper_bundle()` — zip 打包/解包论文数据（含路径安全检查）
- `get_preferences()` / `put_preferences()` — 偏好 CRUD
- `get_profile()` / `put_profile()` — 画像 CRUD

### 修改后端文件

- `backend/main.py` — 注册 sync 路由 (`/api/sync`)
- `backend/config.py` — 新增固定云端地址 `sync_url` 和设备同步凭证 `sync_token`
- `backend/services/arxiv_service.py` — `_save_meta()` 自动写入 `updated_at`
- `backend/services/storage_service.py` — `save_chat_history()` 和 `save_cross_paper_chat_history()` 自动写入 `updated_at`

### 同步角色（当前主路径）

- **`sync_role=client`**：本地后端（Electron 启动的内置后端、`start-cursor-mode.sh` 拉起的本地后端）。它会主动同步：启动 background worker、接收本地写操作触发、定期轮询云端 manifest。
- **`sync_role=server`**：云端后端（VPS / Nginx 反代后的正式服务）。它只作为被动 Sync API 服务端，响应 `/api/sync/*`，不启动后台 worker，也不在论文增删/偏好更新后自动 `request_sync`。
- **`sync_role=off`**：预留给禁用主动同步的特殊环境；保留 `/api/sync/*` 数据能力，但不做任何客户端式行为。

角色通过后端配置显式决定，不再依赖 `127.0.0.1` / `localhost` 这类环境猜测。

### 本地端同步流程（当前主路径）

```
本地后端启动时 → 启动 background sync worker
  ↓
本地写操作（加论文 / 删论文 / 改偏好 / 应用画像）→ 2 秒防抖推送
  ↓
每 15 秒主动轮询云端 manifest
  ↓
先比较 tombstone：哪边的删除时间更晚，就把删除传播到另一边
  ↓
云端有更新 → GET /sync/papers/{id}/bundle → 写入本地
本地有更新 → PUT /sync/papers/{id}/bundle → 推到云端
  ↓
同步 preferences 和 profile（同理对比 updated_at）
```

同步时机：
- 本地后端启动时自动拉起 background sync worker
- 本地写操作触发 2 秒防抖推送
- 每 15 秒主动轮询一次云端 manifest，用于拉取其他端的新内容
- Electron 启动时仍会做一次补同步，但它不再是唯一同步来源

### 配置边界（非常重要）

- **固定云端**：当前产品假设只有一个云端，默认固定为 `https://www.moshang.xyz/ipaper/api`
- **显式角色**：`sync_role` 明确区分本地同步客户端与云端被动服务端；云端不会再因为监听 `127.0.0.1` 就被误判成“本地模式”
- **本地配置**：当前 Electron 设置页里展示和修改的是本地后端配置，例如本地邀请码、设备同步凭证
- **云端配置**：云端 Web 页面里的管理员配置只作用于服务器端，不会自动回写到本地
- **参与同步的数据**：论文 bundle、偏好（`preferences.json`）、用户画像（`profile.md`）
- **不参与同步的配置**：邀请码、LLM API Key、`hjfy_cookie`、固定云端地址、设备同步凭证

### 为什么把同步下沉到本地后端

- 仅把同步循环放在 `electron/main.js` 中存在运行态缺陷：旧 Electron 壳可能继续挂着，但实际端口 `3000/5173` 已经被后来启动的本地服务接管
- 在这种“老壳 + 新服务”的混合态里，用户明明看起来在 Electron app 中操作，但主进程里的同步逻辑未必还对应当前这套前后端
- 过去 Electron 主进程和本地后端都各自实现过一套主动同步客户端逻辑，容易出现重复轮询、状态不一致和排障困难
- 现在改为**只保留本地后端这一份主动同步客户端实现**；Electron 主进程只负责拉起本地后端和显示窗口，不再自己轮询/推送

### 为什么云端必须保持被动

- 云端服务器是同步的**中心服务端**，它的职责是暴露 `/api/sync/*` 并存储数据，而不是再把自己当成一台需要主动同步的“本地设备”
- 如果云端也启动客户端式 background worker，就可能出现“服务器同步自己”或角色混淆：同一套后端同时扮演中心服务端和主动客户端，未来一旦配置了上游地址或凭证，排查会非常困难
- 当前云端默认不配置 `sync_token`，因此保留主动客户端代码没有实际收益，只会增加部署复杂度和误触发风险

### 服务器部署与清理注意事项

- 服务器仓库必须保持干净，避免未提交的本地分叉阻塞 `git pull`
- 若服务器上出现 `._*`（AppleDouble）文件，说明曾有 macOS 文件直接拷贝到服务器；这些属于垃圾文件，应在备份后清理
- 若服务器上存在“内容已等于远端提交、但仍显示 modified”的 tracked 文件，应在备份 `git diff` 后恢复到仓库状态，避免后续每次部署都被拦住

### 当前同步凭证模型

- `sync_token` 现已升级为**设备级同步凭证**，不再复用网页登录 JWT
- 它只允许访问云端 `/api/sync/*`
- 凭证通过云端管理员接口签发，和网页登录态分离，可单独吊销某台设备
- 当前默认**不设置过期时间**；安全性依赖“设备粒度 + 可撤销”，而不是短期过期

## 注意事项

- 服务器 472MB 内存，sync bundle 传输大 PDF 时注意内存峰值
- `bcrypt` 版本锁定 4.2.1（和 passlib 1.7.4 兼容），见 `requirements.txt`
- 本地模式免认证只对 `Host=127.0.0.1/localhost` 的回环请求生效；经 Nginx 暴露的公网云端请求必须走 JWT 或设备同步凭证
- 前端 `authFetch` 已处理 401 自动登出，离线代理层需要在 `authFetch` 之上再包一层
