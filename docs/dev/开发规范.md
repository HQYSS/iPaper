# iPaper 开发规范

> 本文档面向所有参与 iPaper 开发的 AI Agent。
> 目标：确保每个 agent 能快速入手、安全开发、完整交接，形成闭环。

---

## 一、入手阶段：新会话的第一件事

### 1.1 必读文档（按顺序）

1. **本文件**（`开发规范.md`）— 了解规则
2. **`当前状态与待办.md`** — 了解项目当前状态、已知问题、待开发功能
3. **`00-项目概览.md`** — 了解技术栈、架构、目录结构

只有在任务涉及具体子系统时，才需要进一步阅读对应的参考文档（01-04）。

### 1.2 了解边界

在开始工作前，必须清楚：
- 当前任务涉及哪些文件和模块
- 是否涉及服务器操作（见下方"高风险操作"）
- 是否会影响已有功能

## 二、开发阶段：怎么干活

### 2.1 高风险操作——必须征求用户确认

以下操作**在执行前必须告知用户并获得明确确认**：

| 类型 | 示例 |
|------|------|
| **服务器操作** | 修改 Nginx 配置、重启服务、安装系统级包、修改 Xray/VPN 相关配置 |
| **数据操作** | 删除/覆盖用户数据（论文、对话历史、用户画像）|
| **架构变更** | 改变 API 接口格式（破坏前后端兼容）、改变数据存储结构、改变部署方式 |
| **第三方依赖** | 引入新的重要依赖、升级可能有 breaking change 的依赖 |
| **Git 操作** | force push、删除分支、修改已 push 的 commit |

**低风险操作可以直接执行**：修改代码文件、读取配置、运行测试、git add/commit/push（非 force）。

### 2.2 提问而非假设

当需求有歧义时，**问用户**，不要自己猜。特别是：
- 用户说"部署到服务器"但没说路径 → 问
- 功能有多种实现方式且各有取舍 → 问
- 不确定某个改动是否会影响现有功能 → 问

### 2.3 服务器操作规程

1. **只读先行**：先用只读命令检查环境，理解现状
2. **备份再改**：修改配置文件前先 `cp xxx xxx.bak`
3. **测试再生效**：Nginx 改配置后先 `nginx -t`，测试通过才 reload
4. **不碰 Xray**：443 端口由 Xray 管理，绝对不要修改 Xray 配置或重启 Xray

### 2.4 代码风格

| 项目 | 规范 |
|------|------|
| 前端语言 | TypeScript，严格模式 |
| 后端语言 | Python 3.9+，类型标注 |
| 缩进 | 前端 2 空格，后端 4 空格 |
| 注释 | 只写解释"为什么"的注释，不写描述"做了什么"的注释 |
| 命名 | 前端 camelCase，后端 snake_case |
| 组件 | React 函数组件 + Hooks |
| 状态管理 | Zustand |
| 样式 | Tailwind CSS，不写自定义 CSS 除非 Tailwind 无法实现 |

### 2.5 部署同步

代码改动后需要同步到服务器时：

```bash
# 1. 本地 commit & push
git add -A && git commit -m "..." && git push

# 2. 服务器 pull
ssh aws "cd ~/iPaper && git pull"

# 3. 如果改了前端，本地构建后上传（服务器内存不够 build）
cd frontend && npm run build
tar czf /tmp/dist.tar.gz -C frontend dist/
scp /tmp/dist.tar.gz aws:/tmp/
ssh aws "cd ~/iPaper/frontend && tar xzf /tmp/dist.tar.gz && rm /tmp/dist.tar.gz"

# 4. 如果改了后端，重启服务
ssh aws "sudo systemctl restart ipaper-backend"

# 5. 如果改了 nginx
ssh aws "sudo /usr/sbin/nginx -t && sudo /usr/sbin/nginx -s reload"
```

## 三、收尾阶段：文档维护闭环

### 3.1 每次会话结束前，必须做以下检查

| 检查项 | 操作 |
|--------|------|
| **更新 `当前状态与待办.md`** | 新完成的功能加到"已完成"；发现的 bug 加到"已知问题"；新的待办加到"待开发" |
| **更新相关参考文档** | 改了技术栈/目录结构/数据格式 → 更新 00；改了前端架构 → 更新 01；改了后端 API → 更新 02；改了画像系统 → 更新 04；以此类推 |
| **Git commit** | 有意义的 commit message，概括做了什么和为什么 |

### 3.2 文档更新原则

- **保持同步**：文档描述的内容必须和代码一致。如果改了代码但没改文档，后面的 agent 会被误导
- **增量更新**：在现有文档上修改/追加，不要重写整个文件（除非结构需要重组）
- **人类可读**：文档是写给人（和 agent）看的，不是给编译器看的。用清晰的中文，必要时用表格和图

### 3.3 什么时候需要新建文档

当出现以下情况时，应在 `docs/dev/` 下新建编号文档：
- 新增了一个独立的子系统（如翻译系统、用户认证系统）
- 某个模块的复杂度大到无法用已有文档的一个小节覆盖

## 四、项目特有注意事项

### 4.1 用户画像是核心资产

`~/.ipaper/user_profile/profile.md` 是经过对齐访谈精心构建的，任何修改都需要经过自进化 pipeline 的验证流程，不能直接覆盖。

### 4.2 LLM API Key 不入库

`~/.ipaper/config.json` 含 API Key，不在 git 仓库中。部署到新环境时需手动复制。

### 4.3 服务器环境限制

- 内存 472MB，不能在服务器上执行 `npm run build` 或其他内存密集操作
- Python 3.9.2（非最新，注意语法兼容）
- Xray 占据 443 端口，Nginx 在 8080（localhost），部署的 web 应用通过 Nginx location 提供
