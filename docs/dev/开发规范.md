# iPaper 开发规范

> 本文档面向所有参与 iPaper 开发的 AI Agent。
> 目标：确保每个 agent 能快速入手、安全开发、完整交接，形成闭环。

---

## 一、入手阶段：新会话的第一件事

### 1.1 必读文档（按顺序）

1. **本文件**（`开发规范.md`）— 了解规则
2. **`00-项目概览.md`** — 了解技术栈、架构、目录结构
3. **`当前状态与待办.md`** — 了解项目当前状态、已知问题、待开发功能

只有在任务涉及具体子系统时，才需要进一步阅读对应的参考文档（01-04）。

### 1.2 了解边界

在开始工作前，必须清楚：
- 当前任务涉及哪些文件和模块
- 是否涉及服务器操作（见下方"高风险操作"）
- 是否会影响已有功能

## 二、开发阶段：怎么干活

### 2.1 高风险操作——必须征求用户确认

以下操作**在执行前必须告知用户并获得明确确认**：

| 类型 | 示例 |
|------|------|
| **服务器操作** | 修改 Nginx 配置、重启服务、安装系统级包、修改 Xray/VPN 相关配置 |
| **数据操作** | 删除/覆盖用户数据（论文、对话历史、用户画像）|
| **架构变更** | 改变 API 接口格式（破坏前后端兼容）、改变数据存储结构、改变部署方式 |
| **第三方依赖** | 引入新的重要依赖、升级可能有 breaking change 的依赖 |
| **Git 操作** | force push、删除分支、修改已 push 的 commit |

**低风险操作可以直接执行**：修改代码文件、读取配置、运行测试、git add/commit/push（非 force）。

### 2.1.1 能自己做的就自己做

Agent **能独立完成的操作不要等用户来做**，除非该操作属于上表的高风险类型。例如：

- 改完前端代码后需要重启 `npm run dev` → **直接自己重启**，不要说"请你重启前端"
- 改完后端代码后需要重启本地 uvicorn → **直接自己重启**
- 需要部署到服务器 → 按部署同步流程自行执行（服务器操作已在高风险表中，需确认的除外）
- 需要安装 npm/pip 依赖 → **直接安装**

反面例子：让用户手动执行 `npm run dev`、`pip install`、`git push` 等 Agent 完全有能力自行完成的操作。

### 2.2 先讨论后动手

**在用户明确说"开始改"之前，不得执行任何代码修改。** 正确的流程是：

1. 分析任务，汇总所有需要改动的内容（涉及哪些文件、改什么、为什么）
2. 向用户完整呈现方案，等待讨论和确认
3. 用户明确同意后，才开始执行修改

即使方案看起来很明确、没有歧义，也必须等用户确认。Agent 的职责是**提供方案**，不是**替用户做决定**。

### 2.2.1 能自测的先自测

对前端交互类问题，Agent **不能只依赖静态阅读代码推断**。只要环境允许，就应优先自行完成一轮最小可行自测，再请用户验收。

推荐顺序：

1. 先阅读代码，形成问题假设
2. 能运行前端就运行前端，能连浏览器/Electron 就实际操作复现
3. 修改后先自行验证核心路径，确认现象确实消失
4. 自测完成后，再请用户做验收测试

**Cursor 自带浏览器**（`browser-use` subagent 或 Simple Browser `http://localhost:5173`），不要让用户替你做可以自己完成的验证。优先自行启动前后端、打开浏览器测试交互，确认无误后再请用户验收。

说明：
- `npm run build`、lint、单元测试只能证明"没有明显编译/类型错误"，**不能替代前端交互自测**
- 对滚动位置、面板切换、快捷键、拖拽、焦点、动画、异步加载时序等问题，应优先做真实交互验证
- 如果当前环境无法完成自测，要明确告诉用户"哪些已经验证，哪些还无法验证"，不要把未验证内容表述成已确认修复

### 2.3 提问而非假设

当需求有歧义时，**问用户**，不要自己猜。特别是：
- 用户说"部署到服务器"但没说路径 → 问
- 功能有多种实现方式且各有取舍 → 问
- 不确定某个改动是否会影响现有功能 → 问

### 2.3 服务器操作规程

1. **只读先行**：先用只读命令检查环境，理解现状
2. **备份再改**：修改配置文件前先 `cp xxx xxx.bak`
3. **测试再生效**：Nginx 改配置后先 `nginx -t`，测试通过才 reload
4. **不碰 Xray**：443 端口由 Xray 管理，绝对不要修改 Xray 配置或重启 Xray

### 2.4 代码风格

| 项目 | 规范 |
|------|------|
| 前端语言 | TypeScript，严格模式 |
| 后端语言 | Python 3.9+，类型标注 |
| 缩进 | 前端 2 空格，后端 4 空格 |
| 注释 | 只写解释"为什么"的注释，不写描述"做了什么"的注释 |
| 命名 | 前端 camelCase，后端 snake_case |
| 组件 | React 函数组件 + Hooks |
| 状态管理 | Zustand |
| 样式 | Tailwind CSS，不写自定义 CSS 除非 Tailwind 无法实现 |

### 2.5 部署同步

代码改动后需要同步到服务器时：

```bash
# 1. 本地 commit & push
git add -A && git commit -m "..." && git push

# 2. 服务器 pull
ssh aws "cd ~/iPaper && git pull"

# 3. 如果改了前端，本地构建后上传（服务器内存不够 build）
cd frontend && npm run build
tar czf /tmp/dist.tar.gz -C frontend dist/
scp /tmp/dist.tar.gz aws:/tmp/
ssh aws "cd ~/iPaper/frontend && tar xzf /tmp/dist.tar.gz && rm /tmp/dist.tar.gz"

# 4. 如果改了后端，重启服务
ssh aws "sudo systemctl restart ipaper-backend"

# 5. 如果改了 nginx
ssh aws "sudo /usr/sbin/nginx -t && sudo /usr/sbin/nginx -s reload"
```

## 三、收尾阶段：文档维护闭环

### 3.0 前提：用户验收通过

**文档更新和 Git commit 必须在用户确认改动验收通过之后才能执行。** 开发完成后，Agent 应等待用户测试和反馈。只有用户明确表示通过（或要求提交），才进入下面的文档同步流程。

如果用户反馈有问题，回到开发阶段继续修改，不要更新文档。

### 3.1 每次会话结束前，必须执行文档同步 checklist

**步骤 1：列出本次改动涉及的文件和模块。**

**步骤 2：逐条核对下表，打勾或跳过。不要凭记忆判断"已经更新过了"，必须实际打开文档确认内容与代码一致。**

| 改了什么 | 需要更新的文档 | 更新什么 |
|----------|---------------|----------|
| 任何功能完成/发现 bug/新增待办 | `当前状态与待办.md` | 对应分类加一行 |
| 技术栈、目录结构、数据格式、启动方式 | `00-项目概览.md` | 保持示例代码/配置与实际一致 |
| 前端组件、Store、样式、API 封装 | `01-前端.md` | 新组件加文档、改了 Props/功能要同步 |
| 后端服务、API 接口、数据模型、配置 | `02-后端.md` | 服务描述、接口表、模型字段要同步 |
| Electron、iPaper.app、部署流程 | `03-Electron与部署.md` | 启动流程、目录结构、网络连接要同步 |
| 用户画像系统、自进化 pipeline | `04-用户画像与自进化.md` | pipeline 步骤、API、配置要同步 |

**步骤 3：Git commit。** 有意义的 commit message，概括做了什么和为什么。

### 3.2 文档更新原则

- **保持同步**：文档描述的内容必须和代码一致。如果改了代码但没改文档，后面的 agent 会被误导
- **增量更新**：在现有文档上修改/追加，不要重写整个文件（除非结构需要重组）
- **人类可读**：文档是写给人（和 agent）看的，不是给编译器看的。用清晰的中文，必要时用表格和图

### 3.3 什么时候需要新建文档

当出现以下情况时，应在 `docs/dev/` 下新建编号文档：
- 新增了一个独立的子系统（如翻译系统、用户认证系统）
- 某个模块的复杂度大到无法用已有文档的一个小节覆盖

## 四、双端开发规则

iPaper 同时运行在 Electron 桌面端和 Web 端，代码只有一套，但运行环境有差异。以下规则确保双端一致性。

### 4.0 环境判断

**统一使用 `services/env.ts`**，不要在组件中散落 `window.electronAPI` 判断。

```typescript
import { env } from '../services/env'

if (env.isElectron) { /* Electron 专属逻辑 */ }
if (env.isWeb) { /* Web 端专属逻辑 */ }
```

### 4.1 Store 层统一使用离线增强函数

**所有 Store 的 API 调用统一使用 `offlineApi.ts` 的 `xxxOffline()` 函数**，不区分 Web/Electron。

| Store | 使用 | 不使用 |
|-------|------|--------|
| paperStore | `fetchPapersOffline()` | `api.fetchPapers()` |
| chatStore | `getChatHistoryOffline()`, `listSessionsOffline()`, ... | `api.getChatHistory()`, ... |
| preferencesStore | `getPreferencesOffline()`, `updatePreferencesOffline()` | `api.getPreferences()`, ... |
| PdfViewer | `fetchPdfBlobOffline()` | `authFetch(getPdfUrl(...))` |

对 Electron 端来说后端总是在线的，network-first 策略退化为纯 network 调用，零额外开销。

### 4.2 Sync API 向后兼容

修改 `/api/sync/*` 端点时，必须**向后兼容**。如果是 breaking change，必须**先部署服务器端，再更新 Electron 端**。

### 4.3 数据格式兼容

修改 `meta.json`、对话 JSON、`sessions.json` 等数据文件格式时：
- **新增字段**：代码中必须兼容该字段缺失的情况（旧数据没有这个字段）
- **不删除已有字段**：只做增量添加
- **不改变已有字段的类型**

### 4.4 Service Worker 版本管理

修改 `frontend/public/sw.js` 中的缓存逻辑后，**必须递增 `CACHE_NAME` 版本号**（如 `ipaper-static-v1` → `ipaper-static-v2`），否则用户浏览器不会更新缓存策略。

### 4.5 部署顺序

| 改了什么 | 部署顺序 |
|---------|---------|
| 只改前端 | 本地 build → 上传 dist 到服务器 |
| 只改后端 | push → 服务器 pull + restart |
| 改了 Sync API | **先部署服务器**（被同步方），再更新 Electron |
| 改了缓存逻辑 | 递增 SW 版本号 → build → 部署 |

## 五、项目特有注意事项

### 5.1 用户画像是核心资产

`~/.ipaper/user_profile/profile.md` 是经过对齐访谈精心构建的，任何修改都需要经过自进化 pipeline 的验证流程，不能直接覆盖。

### 5.2 LLM API Key 不入库

`~/.ipaper/config.json` 含 API Key，不在 git 仓库中。部署到新环境时需手动复制。

### 5.3 服务器环境限制

- 内存 472MB，不能在服务器上执行 `npm run build` 或其他内存密集操作
- Python 3.9.2（非最新，注意语法兼容）
- Xray 占据 443 端口，Nginx 在 8080（localhost），部署的 web 应用通过 Nginx location 提供
