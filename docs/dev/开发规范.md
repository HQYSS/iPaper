> 本文档是按需查阅的参考手册。核心开发规则已通过 `.cursor/rules/dev-core.mdc` 自动注入每次会话，无需手动阅读。
>
> 本文档包含：代码风格、服务器操作规程、双端开发规则、项目特有注意事项等细节。涉及对应领域时再来查阅。

## 一、代码风格


| 项目          | 规范                                                                                             |
| ----------- | ---------------------------------------------------------------------------------------------- |
| 前端语言        | TypeScript，严格模式                                                                                |
| 后端语言        | Python 3.9+，类型标注                                                                               |
| 缩进          | 前端 2 空格，后端 4 空格                                                                                |
| 注释          | 只写解释"为什么"的注释，不写描述"做了什么"的注释                                                                     |
| 命名          | 前端 camelCase，后端 snake_case                                                                     |
| 组件          | React 函数组件 + Hooks                                                                             |
| 状态管理        | Zustand                                                                                        |
| 样式          | Tailwind CSS，不写自定义 CSS 除非 Tailwind 无法实现                                                        |
| 正则          | **禁止使用 lookbehind** `(?<!...)` / `(?<=...)`（iOS 15.x Safari 不支持，会直接崩溃）                         |
| Markdown 渲染 | 桌面用 ReactMarkdown + rehype-katex；触屏用 `marked` + KaTeX MathML（ReactMarkdown 在 iPad Safari 上会崩溃） |

## 二、服务器操作规程

1. **只读先行**：先用只读命令检查环境，理解现状
2. **备份再改**：修改配置文件前先 `cp xxx xxx.bak`
3. **测试再生效**：Nginx 改配置后先 `nginx -t`，测试通过才 reload
4. **不碰 Xray**：443 端口由 Xray 管理，绝对不要修改 Xray 配置或重启 Xray
5. **Git 工作树必须干净**：服务器部署前先检查 `git status --short`；若存在历史手工改动，先备份 `git diff`，再清理 tracked 脏文件和 `.`_* 这类 AppleDouble 垃圾文件，避免 `git pull` 被拦住

## 三、部署同步（手动参考）

> 日常部署请直接执行 `./scripts/deploy.sh`，它会自动检测并完成以下步骤。
> 以下命令仅供理解部署流程或排查问题时参考。

```bash
# 1. 本地 commit & push
git add -A && git commit -m "..." && git push

# 2. 服务器 pull
ssh aws "cd ~/iPaper && git pull"

# 3. 如果改了前端，本地构建后上传（服务器内存不够 build）
cd frontend && npm run build
tar czf /tmp/dist.tar.gz -C frontend dist/
scp /tmp/dist.tar.gz aws:/tmp/
ssh aws "cd ~/iPaper/frontend && tar xzf /tmp/dist.tar.gz && rm /tmp/dist.tar.gz"

# 4. 如果改了后端，重启服务
ssh aws "sudo systemctl restart ipaper-backend"

# 5. 如果改了 nginx
ssh aws "sudo /usr/sbin/nginx -t && sudo /usr/sbin/nginx -s reload"
```

## 四、双端开发规则

iPaper 同时运行在 Electron 桌面端和 Web 端，代码只有一套，但运行环境有差异。以下规则确保双端一致性。

### 4.0 环境判断

**统一使用 `services/env.ts`**，不要在组件中散落 `window.electronAPI` 判断。

```typescript
import { env } from '../services/env'

if (env.isElectron) { /* Electron 专属逻辑 */ }
if (env.isWeb) { /* Web 端专属逻辑 */ }
```

### 4.1 Store 层统一使用离线增强函数

**所有 Store 的 API 调用统一使用 `offlineApi.ts` 的 `xxxOffline()` 函数**，不区分 Web/Electron。


| Store            | 使用                                                      | 不使用                         |
| ---------------- | ------------------------------------------------------- | --------------------------- |
| paperStore       | `fetchPapersOffline()`                                  | `api.fetchPapers()`         |
| chatStore        | `getChatHistoryOffline()`, `listSessionsOffline()`, ... | `api.getChatHistory()`, ... |
| preferencesStore | `getPreferencesOffline()`, `updatePreferencesOffline()` | `api.getPreferences()`, ... |
| PdfViewer        | `fetchPdfBlobOffline()`                                 | `authFetch(getPdfUrl(...))` |


对 Electron 端来说后端总是在线的，network-first 策略退化为纯 network 调用，零额外开销。

### 4.2 Sync API 向后兼容

修改 `/api/sync/`* 端点时，必须**向后兼容**。如果是 breaking change，必须**先部署服务器端，再更新 Electron 端**。

### 4.3 数据格式兼容

修改 `meta.json`、对话 JSON、`sessions.json` 等数据文件格式时：

- **新增字段**：代码中必须兼容该字段缺失的情况（旧数据没有这个字段）
- **不删除已有字段**：只做增量添加
- **不改变已有字段的类型**

### 4.4 Service Worker 版本管理

修改 `frontend/public/sw.js` 中的缓存逻辑后，**必须递增 `CACHE_NAME` 版本号**（如 `ipaper-static-v1` → `ipaper-static-v2`），否则用户浏览器不会更新缓存策略。

### 4.5 部署顺序


| 改了什么        | 部署顺序                                 |
| ----------- | ------------------------------------ |
| 只改前端        | 本地 build → 上传 dist 到服务器              |
| 只改后端        | push → 服务器 pull + restart            |
| 改了 Sync API | **先部署服务器**（被同步方），再更新 Electron        |
| 改了缓存逻辑      | 递增 SW 版本号 → build → 部署               |
| 服务器上出现本地分叉  | 先备份服务器 `git diff` → 清理脏工作树 → 再执行正常部署 |


### 4.6 本地启动链约束

- **一次只保留一条本地启动链**：手动 `uvicorn/vite`、`start-cursor-mode.sh`、双击 `iPaper.app` 三者不要混用；切换方式前先把旧进程关干净
- **改了 `electron/main.js` / 启动脚本后必须完整重启 app**：只热更前后端服务不会替换已常驻的 Electron 主进程，必须重新启动 `iPaper.app` 或对应启动脚本
- **排查卡顿先看进程再看代码**：先检查是否存在旧 Electron、旧后端、重复 Vite；多实例混跑会让你看到“界面是新版、主进程逻辑还是旧版”的假象

### 4.7 故障排查顺序

遇到以下问题时：没输出、长时间转圈、无响应、状态错乱、切换页面/论文后异常、请求超时、"明明点了但像没生效"。

**必须先看日志，再看代码。不要跳过这一步。**

推荐排查顺序：

1. **先确认实际运行链路**
  - 当前是 Electron、Web、还是手动 `uvicorn + vite`
  - 是否混跑了多个后端 / 多个 Vite / 多个 Electron 实例
  - 当前界面连接到的到底是哪一个后端进程
2. **先看运行时证据**
  - 终端日志：前端 dev server、Electron 主进程、后端日志
  - 浏览器/Electron DevTools：Console、Network、报错堆栈、请求状态码、SSE/流式响应是否有首包
  - 必要时先复现一遍并记录"点击了什么、出现了什么、日志对应出现了什么"
3. **最后才看代码**
  - 只有在日志已经表明问题大致落在哪一层后，才进入代码定位
  - 没看日志前，代码分析只能算假设，不能直接当结论
4. **汇报时区分证据与推测**
  - 哪些结论来自日志/请求记录
  - 哪些只是基于代码的推测

反面例子：

- 用户反馈"一直转圈没输出"，直接去读 Store/组件代码，然后下结论说是前端状态 bug

正确做法：

- 先看当前运行进程和终端日志
- 再看 Network 里请求是否真的发出、是否收到响应、卡在哪一层
- 最后再回到代码解释根因

## 五、项目特有注意事项

### 5.1 用户画像是核心资产

`~/.ipaper/user_profile/profile.md` 是经过对齐访谈精心构建的，任何修改都需要经过自进化 pipeline 的验证流程，不能直接覆盖。

### 5.2 LLM API Key 不入库

`~/.ipaper/config.json` 含 API Key，不在 git 仓库中。部署到新环境时需手动复制。

### 5.3 服务器环境限制

- 内存 472MB，不能在服务器上执行 `npm run build` 或其他内存密集操作
- Python 3.9.2（非最新，注意语法兼容）
- Xray 占据 443 端口，Nginx 在 8080（localhost），部署的 web 应用通过 Nginx location 提供

## 六、浏览器自动化自测（Playwright）

前端改动的强制自测路径。**Agent 自己就能跑通**，不要依赖用户肉眼验证、也不要依赖 `browser-use` subagent（本环境它经常拿不到浏览器工具）。

### 7.1 一次性安装

```bash
mkdir -p /tmp/ipaper-test && cd /tmp/ipaper-test
npm init -y >/dev/null 2>&1
npm install playwright
npx playwright install chromium    # ~92MB，约 2 分钟
```

安装后 `/tmp/ipaper-test/` 和 `~/Library/Caches/ms-playwright/` 会保留，后续 agent 可复用，不用重装。

### 7.2 脚本模板（写到 `/tmp/ipaper-test/test.js` 然后 `node test.js` 跑）

```javascript
const { chromium } = require('playwright')
const fs = require('fs')

const SHOTS = '/tmp/ipaper-test/shots'
fs.mkdirSync(SHOTS, { recursive: true })

;(async () => {
  const browser = await chromium.launch({ headless: true })
  const ctx = await browser.newContext({ viewport: { width: 1440, height: 900 } })
  const page = await ctx.newPage()

  const consoleMsgs = []
  const netErrors = []
  page.on('console', (m) => consoleMsgs.push(`[${m.type()}] ${m.text()}`))
  page.on('pageerror', (e) => consoleMsgs.push(`[pageerror] ${e.message}`))
  page.on('response', (r) => {
    if (r.status() >= 400) netErrors.push(`${r.status()} ${r.request().method()} ${r.url()}`)
  })

  // localhost 免登录模式，直接进主界面
  await page.goto('http://127.0.0.1:5173/', { waitUntil: 'domcontentloaded' })
  await page.waitForTimeout(1500)
  await page.screenshot({ path: `${SHOTS}/01-loaded.png` })

  // 用 title / placeholder / text 做定位，避免脆弱的 class 选择器
  await page.click('button[title="添加论文"]')
  await page.locator('input[placeholder*="arXiv"]').fill('1908.10084')
  await page.locator('button:has-text("添加论文")').first().click()

  // 等待 UI 到达目标状态
  await page.waitForFunction(
    () => !document.body.textContent.includes('正在下载英文 PDF'),
    { timeout: 60000 }
  )
  await page.screenshot({ path: `${SHOTS}/02-ready.png` })

  // 清理测试数据
  await page.evaluate(() => fetch('/api/papers/1908.10084', { method: 'DELETE' }))

  console.log('console:', consoleMsgs.length, 'netErrors:', netErrors)
  await browser.close()
})().catch((e) => { console.error(e); process.exit(1) })
```

### 7.3 验证观察到的 UI

截图保存后用 `Read` 工具读 PNG（Read 支持图像），不要让用户帮你看：

```
Read /tmp/ipaper-test/shots/02-ready.png
```

Read 工具会把图像内容返回给 agent，可以直接判断"spinner 是否出现"、"目标文案是否在界面上"、"是否有错位"等。

### 7.4 自测清单

每次前端改动后 Playwright 脚本里至少覆盖：

- 核心操作路径（改了 X 功能 → 真实操作 X）
- `console` 0 error、`pageerror` 0
- 所有请求 `status < 400`
- 关键 UI 状态的截图 + Read 验证
- 测试完清理副作用（新建的论文/会话要删除）

### 7.5 什么时候退回"让用户帮忙看"

只有以下情况可以放弃 Playwright：

- 需要真实登录凭据且 localhost 免登录路径不覆盖该场景
- 测试涉及 Electron 主进程的 native API（Playwright 只能接标准浏览器）
- 需要真实 GPU / 硬件加速渲染判断

退回时必须按 `dev-core.mdc` § 二.4 明确告知用户"以下无法自测，需要你帮忙确认"。

## 七、文档更新原则

- **保持同步**：文档描述的内容必须和代码一致。如果改了代码但没改文档，后面的 agent 会被误导
- **增量更新**：在现有文档上修改/追加，不要重写整个文件（除非结构需要重组）
- **人类可读**：文档是写给人（和 agent）看的，不是给编译器看的。用清晰的中文，必要时用表格和图
- **新建文档时机**：新增独立子系统、或某模块复杂度大到无法用已有文档的一个小节覆盖时，在 `docs/dev/` 下新建编号文档
