# 用户画像与自进化系统

## 概述

iPaper 的核心创新：根据每个用户的知识背景、讲解偏好、研究兴趣，量身定制论文讲解方式。并通过分析用户与 LLM 的对话反馈，自动进化讲解策略。

```
┌──────────────────────────────────────────────────────────┐
│                       讲解流程                            │
│                                                          │
│  System Prompt = 角色定义 + profile.md（去掉元信息）       │
│                     ↓                                    │
│  User Msg #1 = PDF (base64) + "请为我详细讲解这篇论文"    │
│                     ↓                                    │
│  LLM 按用户画像定制讲解 → SSE 流式返回                    │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                       自进化流程                          │
│                                                          │
│  用户点击"分析对话"按钮                                    │
│         ↓                                                │
│  Step 1: 信号提取（Claude Opus 4.6）                      │
│    输入顺序: ① 当前 profile → ② 论文标题+摘要 → ③ 对话    │
│    输出: 结构化反馈信号列表                                │
│         ↓                                                │
│  Step 2: 编辑规划（Claude Opus 4.6）                      │
│    输入: 当前 profile + 信号列表                           │
│    输出: 精确编辑操作列表                                  │
│         ↓                                                │
│  Step 3: 编辑执行（程序化）                                │
│    对 profile.md 执行最小化编辑                            │
│         ↓                                                │
│  Step 4: 验证（程序化）                                    │
│    检查章节完整性、内容未缩短、编辑已写入                   │
│         ↓                                                │
│  存为 pending_updates.json → 前端通知 → 用户确认/拒绝      │
│         ↓ (确认)                                          │
│  更新 profile.md + 追加 changelog.md                      │
└──────────────────────────────────────────────────────────┘
```

## 用户画像结构 (profile.md)

| 章节 | 内容 | 讲解时的作用 |
|------|------|-------------|
| 一、知识背景 | 深度学习、数学、工程能力的掌握程度表格 | 决定哪些概念可以跳过、哪些需要解释 |
| 二、讲解偏好 | 先全局后局部、视角偏好、类比偏好、实验部分处理 | 决定讲解的结构和风格 |
| 三、禁区 | 绝对不要做的事（如使用统计术语、强行类比） | 硬性约束 |
| 四、当前研究兴趣 | 关注的方向、对特定技术路线的态度 | 决定讲解的侧重点 |
| 五、具体示例库 | 好的/不好的讲解方式的实例 | 通过 few-shot 引导 LLM |
| 六、元信息 | 创建时间、更新次数（不放入 system prompt） | 仅供管理 |

## System Prompt 编译

`UserProfileService.compile_system_prompt()` 做以下处理：

1. 读取 `profile.md` 全文
2. 截掉"六、元信息"段（对 LLM 无用）
3. 前面拼接角色定义
4. 包裹为 system prompt

最终结构：
```
你是一个专业的学术论文阅读助手。你的任务是根据用户的知识背景和偏好，为用户量身定制论文讲解。

以下是该用户的详细画像，你必须严格遵循其中的所有要求：

---
[profile.md 内容，不含元信息]
---

请根据以上画像来调整你的讲解方式。
```

## 自进化 Pipeline 详解

### 模型选择

| 用途 | 模型 | 理由 |
|------|------|------|
| 论文讲解 | `gemini-3-pro-preview-thinking` | 长 context、多模态（读 PDF） |
| 画像分析（Step 1 & 2） | `cu-claude-opus-4-6` | 指令跟随最好，适合结构化提取和编辑规划 |

两者共用同一个 `api_base` 和 `api_key`。配置在 `config.py` 的 `ProfileAnalysisConfig` 中。

### Step 1: 信号提取

**目标**：从对话中识别反馈信号，不做任何编辑。

**输入顺序（重要）**：
1. 当前 profile → LLM 先知道已记录了什么，避免重复提取
2. 论文标题 + 摘要 → 建立领域上下文
3. 完整对话记录 → 带着背景去分析

**信号类型**：

| 类型 | 含义 | 示例 |
|------|------|------|
| `knowledge_update` | 知识掌握度变化 | 用户说"这个我知道，跳过" |
| `positive_example` | 好的讲解方式 | 用户说"这个类比好，懂了" |
| `negative_example` | 差的讲解方式 | 用户说"没懂，换种方式" |
| `preference_update` | 讲解风格偏好 | 用户要求"再展开讲讲" |
| `interest_update` | 研究兴趣变化 | 用户提到新的关注方向 |

**输出**：JSON，每个信号含 `type`, `description`, `evidence`, `context`。

### Step 2: 编辑规划

**目标**：将信号转化为对 profile 的精确编辑操作。

**编辑操作类型**：

| 操作 | 说明 | 用途 |
|------|------|------|
| `append_example` | 向示例库追加 | 添加好的/不好的讲解案例 |
| `append_item` | 向指定章节追加内容 | 新增知识点、偏好、禁区等 |
| `replace_text` | 精确查找替换 | 修改已有条目 |

**编辑原则**：
- 最小化编辑，每个信号只改必要的部分
- 示例库优先（大多数反馈应变为示例）
- 不删除已有内容
- 不修改元信息

### Step 3: 编辑执行

程序化应用编辑操作到 profile 文本：
- `append_example`：找到示例库小节，替换占位符或追加到末尾
- `append_item`：找到目标章节标记，在下一个章节之前插入
- `replace_text`：精确查找并替换（仅替换第一个匹配）

### Step 4: 验证

检查编辑后的 profile 完整性：
- 所有 `##` 级章节标题是否都还在
- 内容不能缩短超过 20%
- 每个编辑操作的内容确实出现在新 profile 中

## API 接口

### 画像更新 (`/api/profile`)

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/profile/pending-updates` | 获取待确认的更新（含信号列表和编辑详情） |
| POST | `/api/profile/apply-updates` | 确认并应用更新（写入 profile.md + changelog.md） |
| POST | `/api/profile/reject-updates` | 拒绝更新（删除 pending_updates.json） |
| POST | `/api/profile/trigger-analysis` | 触发画像分析（后台异步执行） |

### 触发方式

当前为**手动触发**：用户点击 ChatPanel 标题栏的分析按钮（RefreshCw 图标）。按钮在对话不足 2 轮时禁用。

## 前端交互

### ChatPanel 标题栏

- 分析按钮（RefreshCw 图标）：点击后触发画像分析，分析期间显示 loading 动画
- 按钮禁用条件：正在分析中 / 对话少于 2 轮

### 画像更新通知

分析完成后，如果有更新建议，在 ChatPanel 顶部显示蓝色通知卡片：
- 摘要文字
- 来源论文
- 可展开查看详情（信号列表 + 编辑操作）
- 应用 / 忽略 按钮

### 自动讲解

论文加载后，如果没有对话历史，自动发送"请为我详细讲解这篇论文。"触发首次讲解。

## Changelog 格式

每次画像更新应用后，自动追加到 `changelog.md`：

```markdown
---

## [2026-02-26] 自动更新（基于论文「DexterityGen: ...」的对话）

**触发方式**：对话反馈自动分析

**变更摘要**：用户在对话中表示对 Inverse Dynamics 的 MLP 实现方式很满意...

**识别到的信号**：
- [positive_example] 用户认为"Autocorrect"类比很好
- [knowledge_update] 用户对 Gradient Guidance 已了解

**执行的编辑**：
- 向好的讲解方式添加了"Autocorrect"类比案例
- 更新了知识背景中 Diffusion Guidance 的熟练度
```

## 部署相关

### 服务器配置

画像数据存储在 `~/.ipaper/user_profile/`，需在部署时复制到服务器。初始画像从 `docs/user_profiles/` 复制：

```bash
scp ~/.ipaper/user_profile/profile.md aws:~/.ipaper/user_profile/
scp ~/.ipaper/user_profile/changelog.md aws:~/.ipaper/user_profile/
```

### 配置项

在 `config.py` 中：

```python
class ProfileAnalysisConfig:
    model: str = "cu-claude-opus-4-6"       # 画像分析专用模型
    temperature: float = 0.2                # 低温度保证稳定输出
    max_tokens: int = 4096
```

与 LLM 讲解模型共用 `api_base` 和 `api_key`。
