# 用户画像与自进化系统

## 概述

iPaper 的核心创新：根据每个用户的知识背景、讲解偏好、研究兴趣，量身定制论文讲解方式。并通过分析用户与 LLM 的对话反馈，人机协作进化讲解策略。

```
┌──────────────────────────────────────────────────────────┐
│                       讲解流程                            │
│                                                          │
│  System Prompt = 角色定义 + profile.md（去掉元信息）       │
│                     ↓                                    │
│  User Msg #1 = PDF (base64) + "请为我详细讲解这篇论文"    │
│                     ↓                                    │
│  LLM 按用户画像定制讲解 → SSE 流式返回                    │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                    自进化流程（人机协作）                   │
│                                                          │
│  用户点击 ChatPanel 标题栏"进化"按钮                      │
│         ↓                                                │
│  中间区域切换为 ProfilePanel（左右分栏）                   │
│  左侧：当前画像渲染   右侧：进化 Agent 对话               │
│         ↓                                                │
│  Agent 自动开始分析对话（Claude Opus 4.6）                │
│  上下文: 论文 PDF + 当前画像 + 对话记录                   │
│         ↓                                                │
│  Agent 展示分析结果 → 用户协商调整 → Agent 输出编辑计划    │
│         ↓                                                │
│  前端解析 <edit_plan> → 编辑计划 diff 预览                │
│         ↓                                                │
│  用户点击"应用到画像" →                                   │
│    后端: 程序化执行编辑 → 验证 → 写入 profile.md          │
│    追加 changelog.md                                     │
│    左侧画像自动刷新                                       │
└──────────────────────────────────────────────────────────┘
```

## 用户画像结构 (profile.md)

| 章节 | 内容 | 讲解时的作用 |
|------|------|-------------|
| 一、知识背景 | 深度学习、数学、工程能力的掌握程度表格 | 决定哪些概念可以跳过、哪些需要解释 |
| 二、讲解偏好 | 先全局后局部、视角偏好、类比偏好、实验部分处理 | 决定讲解的结构和风格 |
| 三、禁区 | 绝对不要做的事（如使用统计术语、强行类比） | 硬性约束 |
| 四、当前研究兴趣 | 关注的方向、对特定技术路线的态度 | 决定讲解的侧重点 |
| 五、具体示例库 | 好的/不好的讲解方式的实例 | 通过 few-shot 引导 LLM |
| 六、元信息 | 创建时间、更新次数（不放入 system prompt） | 仅供管理 |

## 进化 Agent 设计

### 模型

| 用途 | 模型 | 理由 |
|------|------|------|
| 论文讲解 | `google/gemini-3.1-pro-preview` | 长 context、多模态（读 PDF） |
| 进化 Agent | `anthropic/claude-opus-4.6` | 结构化分析和编辑规划能力最强 |

两者共用 `api_base` 和 `api_key`（OpenRouter）。配置在 `config.py` 的 `ProfileAnalysisConfig`，`max_tokens: 16384`。

### Agent 上下文

每次进化对话传给 Claude 的完整内容：

```
[system]  进化 Agent 的 system prompt（身份 + 工作指南）

[user]    第一条消息（multimodal）：
            ① 论文 PDF 原文（base64，单论文或串讲的多篇）
            ② 当前用户画像全文
            ③ 论文标题 + 摘要
            ④ 论文讨论的完整对话记录

[assistant] Agent 分析结果

[user]    用户反馈/调整

...（多轮协商）
```

### System Prompt 核心理念

进化的终极目标：**让讲解模型足够了解用户，以至于用户尽可能不需要追问。** 用户每一次追问都暴露画像的一个缺口。Agent 分析对话时的核心问题不是"用户说了什么"，而是"如果画像里提前写了什么，讲解模型就不会让用户产生这次追问"。

### 信号类型

| 类型 | 含义 |
|------|------|
| `knowledge_update` | 用户的追问暴露了画像中知识背景的不准确 |
| `preference_update` | 用户的追问暴露了画像中对用户关注点的记录不足 |
| `positive_example` | 某种讲解方式效果好，值得记录（需有泛用性） |
| `negative_example` | 某种讲解方式效果差，应避免（需有泛用性） |
| `interest_update` | 用户的选题和追问方向暴露了研究兴趣变化 |

### 编辑操作

| 操作 | 用途 |
|------|------|
| `append_example` | 向示例库（5.1 或 5.2）添加案例 |
| `append_item` | 向指定章节追加内容 |
| `replace_text` | 精确替换画像中的一段文字 |

Agent 拥有完全编辑权（可大改、删除、重组），最终经用户确认才生效。唯一硬性约束：不改元信息（第六章）。

示例库的写入要求**泛用性**：每条记录必须能指导未来多篇论文的讲解，不是记录某篇论文某个概念的正确/错误解释。

### 编辑计划输出

Agent 在用户确认后，用 `<edit_plan>` 标签输出 JSON，后端自动解析。

## 前端交互

### 触发入口

ChatPanel 标题栏的进化按钮（UserCog 图标），单论文和串讲模式都可用。按钮在对话不足 2 轮时禁用。

### ProfilePanel 布局

点击进化按钮后，中间区域从 PdfViewer/CrossPaperViewer 切换为 ProfilePanel（左右五五分栏）：

```
┌──────────┬───────────────┬───────────────┬──────────────┐
│ PaperLib │ 当前画像渲染   │ 进化 Agent    │ ChatPanel    │
│          │ (profile.md)  │ 对话 + 编辑   │ (对话历史)   │
│          │               │ 计划 diff     │              │
└──────────┴───────────────┴───────────────┴──────────────┘
```

- 左半：当前画像 Markdown 渲染，打开时自动加载，应用编辑后自动刷新
- 右半上：进化 Agent 对话（Markdown 渲染）
- 右半中：编辑计划预览（有 edit_plan 时出现，diff 样式）
- 右半下：输入框 + 应用/放弃按钮

### 状态管理

`profileStore.ts`（独立于 chatStore）：

| 状态 | 说明 |
|------|------|
| `isEvolutionOpen` | 面板是否打开 |
| `evolutionMessages` | 进化对话消息列表 |
| `isStreaming` | 流式接收中 |
| `editPlan` | 解析出的编辑计划（edits + changelog_summary） |
| `profileContent` | 当前画像内容（用于左侧渲染） |
| `sourcePaperId` / `sourceCrossPaperSessionId` | 分析来源 |

### 关闭/中断处理

- 关闭面板时：abort SSE 连接、clearTimeout 自动启动 timer
- 打开新面板时：先 abort 旧连接
- 流式处理中检查 `isEvolutionOpen`：面板关了就 break
- SSE 意外断开（服务器重启等）：检测到 stream 结束但没收到 done，显示"连接意外断开，请重试"

## API 接口

### 画像进化 (`/api/profile`)

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/profile/current` | 获取当前画像内容 |
| GET | `/api/profile/changelog` | 获取变更日志 |
| POST | `/api/profile/evolution-chat` | 进化 Agent 对话（SSE 流式） |
| POST | `/api/profile/save-edit-plan` | 保存编辑计划为待确认状态 |
| POST | `/api/profile/apply-updates` | 确认并应用更新 |
| POST | `/api/profile/reject-updates` | 拒绝更新 |
| GET | `/api/profile/pending-updates` | 获取待确认的更新 |

### evolution-chat 请求体

```json
{
  "message": "用户消息",
  "paper_id": "2501.09781",
  "cross_paper_session_id": null,
  "evolution_messages": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

### SSE 流格式

与论文对话相同：`data: {"type": "chunk|done|error", ...}\n\n`

`done` 事件中如果 Agent 输出了 `<edit_plan>`，会额外包含 `edit_plan` 字段。

## 编辑执行与验证

### 程序化执行

根据 `SECTION_MARKERS`（章节标题 → 位置的映射表）定位画像中的目标位置：

- `append_example`：找到示例库小节，替换"暂无"占位符或追加到章节末尾
- `append_item`：找到目标章节标记，在下一个章节之前插入
- `replace_text`：精确查找并替换（仅替换第一个匹配）

### 验证

- 画像不能为空
- 每个编辑操作的内容确实出现在新画像中

### 应用

验证通过后写入 `profile.md`，追加 `changelog.md`，前端左侧画像自动刷新。

## 配置

在 `config.py` 中：

```python
class ProfileAnalysisConfig:
    model: str = "anthropic/claude-opus-4.6"
    temperature: float = 0.2
    max_tokens: int = 16384
```

与 LLM 讲解模型共用 `api_base` 和 `api_key`。
