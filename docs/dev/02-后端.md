## 配置管理 (`config.py`)

使用 `pydantic-settings` 管理配置，支持环境变量（前缀 `IPAPER_`）和配置文件。

```python
class Settings:
    data_dir: Path = ~/.ipaper         # 数据根目录
    host: str = "127.0.0.1"
    port: int = 3000
    llm: LLMConfig                     # LLM 子配置

class LLMConfig:
    provider: str = "openrouter"        # openrouter | cursor_cli
    api_base: str = "https://openrouter.ai/api/v1"
    api_key: str = ""                  # 需要用户配置（OpenRouter API Key）
    model: str = "google/gemini-3.1-pro-preview"
    temperature: float = 0.7
    max_tokens: int = 32768  # Gemini 3.1 Pro 的 reasoning tokens 也计入 max_tokens，留足空间避免正文被截断
    cursor_command: str = "cursor"      # Cursor CLI 命令路径
    cursor_model: str = ""              # 留空使用 Cursor 默认模型
    cursor_timeout_seconds: int = 600

class ProfileAnalysisConfig:
    model: str = "anthropic/claude-opus-4.6"  # 共用 llm 的 api_base 和 api_key
    temperature: float = 0.2
    max_tokens: int = 16384
```

- 配置文件：`~/.ipaper/config.json`
- 全局单例：`settings = Settings()`
- 启动时自动创建 `~/.ipaper/` 和 `~/.ipaper/papers/` 目录
- 调用 `settings.save_config()` 持久化到文件

| `SessionList` | 会话列表 (`sessions[]`, `last_active_session_id`) |

| `CrossPaperAddPapersRequest` | 向串讲添加论文请求 (`paper_ids[]`) |
| `CrossPaperChatRequest` | 串讲对话请求 (`message`, `quotes?`, `page_selections?`) |
| `CrossPaperChatHistory` | 串讲对话历史 (`session_id`, `paper_ids[]`, `messages[]`, `forks?`) |
| `LLMConfigUpdate` | LLM 配置更新请求 |

### 对话 (`/api/chat/{paper_id}/{session_id}`)

| 方法 | 路径 | 说明 | 请求体 | 响应 |
|------|------|------|--------|------|
| POST | `/api/chat/{id}/{sid}` | 发送消息 | `{ message, quotes?: [{text, source}], page_selections?: [{ paper_id?, ranges: [{start, end}] }] }` | SSE 流；若需选页则返回 `409`；若同会话已有正在生成的任务也返回 `409` |
| POST | `/api/chat/{id}/{sid}/stop` | 主动停止当前会话的生成任务 | - | `{ stopped: bool }` |
| GET | `/api/chat/{id}/{sid}/history` | 对话历史 | - | `ChatHistory`（含 `forks`、`draft`） |
| PUT | `/api/chat/{id}/{sid}/history` | 更新历史 | `{ messages, forks? }` | `{ message }`；任务在跑时返回 `409` |
| PUT | `/api/chat/{id}/{sid}/history/draft` | 更新未发送草稿 | `{ draft: { input, quotes?, page_selections? } }` | `{ message }` |
| DELETE | `/api/chat/{id}/{sid}/history` | 清空历史 | - | `{ message }` |

**生成任务的生命周期（重要）：**

LLM 生成任务由 `services/chat_task_service.py` 管理，**和 HTTP 请求生命周期解耦**：

- POST 命中后，路由先把 user 消息 + 空 assistant 占位（`truncated=True`）落盘，再用 `asyncio.create_task` 起一个独立 ChatTask 跑 LLM 流。
- HTTP handler 只是该 ChatTask 的"第一个订阅者"，从内存广播队列里拉 chunk 转 SSE。
- 客户端断连（关 PWA、网络断、`AbortController.abort()`）只会从订阅者列表里摘掉这个 handler，**ChatTask 本身继续跑**，每 0.5s 节流把 partial 写盘（`truncated=True`），结束时再写一次终态（`truncated=False` for `stop`、`True` for `length`/`stopped`/`error`）。
- 用户主动停止必须显式 `POST /stop`，触发 `asyncio.Task.cancel()`。仅 abort fetch 不会停止后端 task。
- 同一 session 不允许并发两个任务；POST/PUT/DELETE 在任务运行时分别返回 409 或先 stop。
- 任务异常终止且没有任何 partial 时，会把 assistant 内容落盘为 `生成失败：...`，避免用户只看到一条没有回复的 user 消息；`GET /history` 只会清理“任务已不存在且没有错误内容”的空 assistant 占位（典型如进程重启、process kill）。
- 任务结束后保留 5 分钟在内存中以应对晚到的订阅者，到点再清理。
- 对话路由和 `ChatTaskService` 会记录关键诊断日志：请求进入、Provider、消息/引用/页码选择数量、PDF 转图、payload 准备、占位落盘、任务启动、订阅者 attach/detach、首个 chunk、终态、空回复或异常。日志写入 `logs/backend.log`。

### 运行版本与日志

- `GET /api/health/runtime` 返回当前后端运行态：PID、启动时间、Python 版本、项目根目录、`sync_role`、LLM Provider、git branch/SHA/dirty 状态。排查“源码已改但行为没变”时先看这个接口。
- `middleware.request_logging.RequestLoggingMiddleware` 为每个 HTTP 请求生成或透传 `X-Request-ID`，响应头也会返回同一个 ID；后端日志会记录 method/path/status/duration/user。
- `services.log_context` 把 request id 和 user id 注入日志格式，业务日志可以通过 `[req=...]` 串起一次点击触发的路由、service、task 日志。
- `POST /api/client-logs` 接收前端和 Electron renderer 侧诊断事件。前端会自动上报全局 JS error、unhandled promise rejection、网络失败、HTTP 5xx、Service Worker 注册失败和离线队列重放失败。
- 本地 `iPaper.app` 启动链仍通过 stdout/stderr 写到 `logs/backend.log`；启动脚本会在拉起后端前记录 git SHA/dirty 状态，并在日志超过 50MB 时轮转为 `.1`。

**SSE 流格式：**

每行 `data: {JSON}\n\n`：

```json
// 任务订阅就绪（前端 fetch 拿到响应后由 services/api.ts 本地合成）
{ "type": "open" }

// 正常响应片段
{ "type": "chunk", "content": "一段文字" }

// 响应因达到 max_tokens 被截断（在 done 之前发出，assistant 消息会标 truncated=true 落盘）
{ "type": "truncated", "reason": "length" }

// 任务正常结束（finish_reason: "stop" | "length"）
{ "type": "done", "finish_reason": "stop" }

// 任务被 POST /stop 主动取消
{ "type": "stopped", "finish_reason": "stopped" }

// 任务异常终止（已映射成中文消息）
{ "type": "error", "finish_reason": "error", "message": "API Key 无效或已过期，请在设置中检查" }
```

当 PDF 已经转成图像但 payload 仍超过 20MB 时，接口不会再默认截取前几页，而是返回 `409`：

```json
{
  "detail": {
    "code": "page_selection_required",
    "message": "《某论文》转成图像后仍超过 20MB，请先选择要保留的页码范围。",
    "requirements": [
      {
        "paper_id": "2511.18870",
        "title": "HunyuanVideo 1.5 Technical Report",
        "total_pages": 42,
        "selected_ranges": [{ "start": 1, "end": 5 }]
      }
    ]
  }
}
```

## 服务层详解

### ArxivService (`services/arxiv_service.py`)

负责论文的下载和本地管理。

**arXiv ID 解析 (`parse_arxiv_input`)：**

支持以下输入格式：
- `2301.12345`
- `2301.12345v1`
- `https://arxiv.org/abs/2301.12345`
- `https://arxiv.org/pdf/2301.12345.pdf`

**普通 PDF URL 解析 (`parse_pdf_url_input`)：**

- 支持 `http://` / `https://` 开头的非 arXiv URL
- arXiv URL 优先走 arXiv metadata 流程，不会被当成普通 PDF URL
- 普通 PDF URL 生成稳定内部 ID：`pdf_<sha256(url)[:16]>`
- `PaperMeta.source_type = "pdf_url"`，`source_url` 保存原始 URL；标题从 URL 文件名推断，作者为空，摘要写入来源 URL

**下载流程 (`download_paper`) — 占位 + 后台下载 + metadata 补齐：**
1. 先解析 arXiv ID；若不是 arXiv，再尝试解析普通 PDF URL
2. 若 `~/.ipaper/data/<user_id>/papers/{id}/meta.json` 已存在：按 `download_status` 分支
   - `ready` → 直接返回已有 meta（去重）
   - `downloading` → 返回现有 meta，并确保后台 task 在跑（进程重启后重新挂起时使用）
   - `failed` → 重置为 `downloading` 重新触发后台下载
3. 新 arXiv 论文：不再同步等待 arXiv metadata API。先写最小占位 meta（标题 `arXiv {id}`、摘要“元数据待补齐”、`download_status="downloading"`），更新 `index.json`，**立即返回**
4. 新普通 PDF URL：不查 arXiv metadata，直接写入占位 meta 并启动下载 task
5. 后台并行启动两个任务：
   - `_download_pdf_async`：下载英文 PDF
   - `_refresh_metadata_async`：仅 arXiv 论文使用，按 arXiv API 规则限速补齐 title/summary/authors/source_url；补齐后更新 `meta.json` 和 `index.json`
6. PDF 下载任务：
   - arXiv 论文：直接下载 `https://arxiv.org/pdf/{id}`，避免下载前再次请求 metadata API；当前环境下 `httpx/requests` 偶发 SSL 读错误，因此失败时回退到标准库 `urllib.request.urlretrieve`
   - 普通 PDF URL：使用浏览器风格 `User-Agent`、`Accept`、`Referer` 通过 `httpx.Client(follow_redirects=True)` 下载，适配部分源站对裸程序请求的拦截
   - 下载内容必须包含 `%PDF` 文件头，成功后原子写：`paper.pdf.part` → rename 到 `paper.pdf`
   - 成功：更新 `meta.json` 为 `download_status="ready"` + 调 `sync_service.request_sync("paper-ready", ...)`
   - 失败：按指数退避重试（2s / 4s / 8s，最多 3 次），仍失败则标记 `download_status="failed"` + 写入 `download_error`
   - 被 cancel（如用户删除论文）直接退出，由 `sync_service.delete_paper` 调 `arxiv_service.cancel_download` 触发

**为什么这样设计：**
- `arxiv.Client().results()` 和 `paper.download_pdf()` 都是同步 blocking 调用，直接在 async 路由里会卡住整个事件循环。用 `to_thread` 丢到线程池后，其它 API（论文列表、对话、翻译状态等）不再受影响。
- 添加按钮不再等待 arXiv metadata API；即使 metadata API 429/超时，PDF 下载仍可先进行，论文 ready 后即可阅读。metadata 晚点补齐只影响列表标题/作者/摘要，不影响 PDF 阅读。
- arXiv metadata 后台任务用单个 `asyncio.Lock` + 3 秒间隔限流，遵守 arXiv legacy API “3 秒 1 次、单连接”的要求，避免频繁添加时主动制造 429。
- 中文翻译仍由 `translation_service.ensure_translation` 独立异步处理，二者互不阻塞。

**索引管理：**
- `index.json` 位于 `~/.ipaper/data/<user_id>/papers/index.json`
- 格式：`{ papers: [{ arxiv_id, title, download_time }] }`
- 添加/删除论文时自动维护；`downloading` 状态也会写进 `index.json`（保持列表顺序稳定）

**PaperMeta 字段：**
- `download_status: "downloading" | "ready" | "failed"`（默认 `ready`，旧 meta 自动兼容）
- `download_error: Optional[str]`（失败原因，供前端展示）
- `source_type: "arxiv" | "pdf_url"`（默认 `arxiv`，旧 meta 自动兼容）
- `source_url: Optional[str]`（arXiv 论文保存 abs URL，普通 PDF 保存原始 PDF URL）

**注意事项：**
- arXiv API 有速率限制（HTTP 429）。metadata 补齐任务会限速；PDF 下载不依赖 metadata API，可绕开“API 不通但 PDF 可直接下载”的场景。
- 普通 PDF URL 仍可能被 Cloudflare/源站策略拒绝（HTTP 403）。后端会记录短错误如“源站拒绝程序下载（HTTP 403）”，但不能保证绕过所有反爬策略。
- **后端进程重启时的中断下载恢复**：启动时 `main.py` 会调用 `arxiv_service.recover_incomplete_downloads()` 扫描所有用户论文目录；若 `meta.json` 停在 `downloading` 但后台 task 已因进程重启消失，会删除残留 `.part` 并重新挂起下载 task。若当前进程里仍有活跃下载 task，则保留正在写入的 `.part`，避免论文列表轮询误删临时文件。若本地 `paper.pdf` 已存在且文件头有效，会直接修正为 `ready`；若 PDF 缺失或损坏，会标记为 `failed` 并提示重新下载。

### LLMService (`services/llm_service.py`)

负责与 LLM Provider 通信。`llm.provider=openrouter` 时走 OpenRouter；`llm.provider=cursor_cli` 时走本机 Cursor CLI headless Agent。

**API 调用方式：**
- 使用 `openai` Python SDK（AsyncOpenAI）
- API 提供商：OpenRouter（`https://openrouter.ai/api/v1`）
- 讲解模型：`google/gemini-3.1-pro-preview`
- Reasoning：通过 `extra_body={"reasoning": {"effort": "medium"}}` 启用 Gemini 思考模式

**Cursor CLI 调用方式：**
- `services/cursor_cli_service.py` 使用 `asyncio.create_subprocess_exec` 调用 `cursor agent -p`
- 参数固定包含 `--output-format stream-json`、`--stream-partial-output`、`--mode ask`、`--trust`、`--workspace <PROJECT_ROOT>`
- 只消费带 `timestamp_ms` 的 assistant 增量文本，跳过最终汇总消息，避免重复输出
- 不传 `--force`，因此默认是只读讲解，不允许 agent 修改文件
- PDF 会先用 PyMuPDF 渲染为缓存 JPEG 页面图（`~/.ipaper/cache/cursor_pdf_pages/`），prompt 同时提供页面图片路径和 PDF 本地路径；Cursor Agent 读取图片路径时走视觉输入，PDF 路径仅作为文本层 fallback
- 单次 Cursor CLI 视觉上下文默认最多 40 页；超过时抛出 `PageSelectionRequiredError`，复用前端页码选择流程
- 页面图默认 `120 DPI`、JPEG quality `75`，兼顾公式/图表可读性与磁盘占用
- Cursor 页面图缓存默认最多 `1024MB`，每次渲染后按文件修改时间清理最旧缓存，避免小 SSD 被页面图长期占满

**PDF 上传方式：**

OpenRouter 模式下，将 PDF 文件编码为 base64，作为 `file` 类型的消息内容发送（OpenRouter 格式）：

```python
{
    "type": "file",
    "file": {
        "filename": "paper.pdf",
        "fileData": "data:application/pdf;base64,{pdf_base64}"
    }
}
```

对于大 PDF：
- `PDF_SIZE_THRESHOLD = 15MB` 时不再发送原始 PDF，而是用 PyMuPDF 逐页渲染 JPEG
- `IMAGE_PAYLOAD_LIMIT = 20MB` 限制的是 JPEG base64 payload 总量
- 若未提供 `page_selections` 且超限，抛出 `PageSelectionRequiredError`
- 若提供的页码范围仍超限，也会再次抛出 `PageSelectionRequiredError`，并把之前的 `selected_ranges` 带回前端，要求继续缩小
- 串讲模式会先遍历所有论文，把所有需要选页的论文一次性收集后统一返回

**消息构建逻辑 (`_build_messages`)：**
1. 添加 system prompt（学术论文阅读助手角色）
2. 在**第一条用户消息**中附加 PDF（base64）
3. 如果有 `quotes`，在最后一条用户消息前拼接引用文本（带来源标注：`[来自论文]` / `[来自对话]`）
4. assistant 消息附加 `reasoning` 字段（如有），传回 API 保持推理连续性
5. 后续消息正常传递

**System Prompt 要点：**
- 使用中文回答，保留英文术语
- 基于论文内容回答，不编造
- 可引用具体章节和图表

**串讲模式 (`chat_stream_cross_paper`)：**
- 使用 `get_cross_paper_system_prompt()` — 在基础用户画像 prompt 后追加串讲专用指令（对比分析框架、`[[id]]` 引用标注格式）
- `_build_cross_paper_first_user_content()` — 在第一条 user 消息中注入所有论文 PDF（每篇前加 `=== 论文 [[id]]: 标题 ===` 标注），复用 `_build_pdf_content_blocks()` 处理大 PDF，并汇总所有需要用户选页的论文
- `_build_messages_cross_paper()` — 构建完整的 API 消息列表

**流式响应 (`chat_stream` / `chat_stream_cross_paper`)：**

返回 `AsyncGenerator[str, None]`，逐 chunk yield 文本。两个可选的「副产物」参数由调用方传入可变容器来收集信息：
- `reasoning_collector: List[str]` — 流式过程中累积 reasoning tokens 片段，调用方在结束后自行 `''.join(...)`
- `finish_reason_holder: dict` — 流结束后写入 `{"value": "stop" | "length" | ...}`，`chat.py` 路由据此判断是否被 `max_tokens` 截断；若为 `"length"` 则向 SSE 发送 `{"type":"truncated","reason":"length"}` 事件，并把 assistant 消息的 `truncated=True` 一起落盘。

### StorageService (`services/storage_service.py`)

管理多会话对话历史的持久化。

**存储位置：** `papers/{arxiv_id}/chats/`

```
chats/
├── sessions.json         # 会话索引
└── {session_id}.json     # 单个会话历史
```

**sessions.json 结构：**
```json
{
  "sessions": [{ "id": "s_170000000", "title": "自动讲解", "created_at": "...", "updated_at": "..." }],
  "last_active_session_id": "s_170000000"
}
```

**会话文件结构：**
```json
{
  "paper_id": "2301.12345",
  "session_id": "s_170000000",
  "messages": [
    { "role": "user", "content": "这篇论文讲了什么?", "quotes": [{ "text": "...", "source": "pdf" }] },
    { "role": "assistant", "content": "这篇论文提出了...", "reasoning": "（可选）" }
  ],
  "forks": { "1": { "alternatives": [[...], [...]], "active": 1 } },
  "draft": {
    "input": "还没发送的问题",
    "quotes": [{ "text": "...", "source": "chat" }],
    "page_selections": [{ "paper_id": "2301.12345", "ranges": [{ "start": 1, "end": 5 }] }]
  }
}
```

**Session 管理方法：**
- `list_sessions(paper_id)` → `SessionList`
- `create_session(paper_id, title?)` → `SessionMeta`
- `delete_session(paper_id, session_id)` → `bool`
- `set_last_active_session(paper_id, session_id)`

**对话历史方法：**
- `get_chat_history(paper_id, session_id)` → `(List[ChatMessage], forks_dict, draft_dict)`
- `save_chat_history(paper_id, session_id, messages, forks?, draft?)` → 覆盖写入
- `clear_chat_history(paper_id, session_id)` → 删除文件

**引用持久化：** 用户消息上的 `quotes` 会跟随 `ChatMessage` 一起写入历史文件；单论文和串讲接口都采用同一结构，前端刷新后可直接恢复引用块显示

**未发送草稿持久化：** 当前会话的输入框文本、待发送引用，以及已经确认过的 `page_selections` 都保存在同一个会话文件的 `draft` 字段中。前端输入过程中会调用 `/history/draft` 接口同步；切换会话和刷新后通过 `GET /history` 恢复；真正发送消息时会把 `draft` 清空，但会把本轮使用的 `page_selections` 一并写回，供同一会话后续继续复用

**向后兼容：** 首次访问论文时，旧 `chat_history.json` 自动懒迁移到 `chats/` 目录

**串讲存储：** 独立于单论文，位于 `~/.ipaper/cross-paper/`：
- `sessions.json`：串讲会话索引（每个 session 额外存 `paper_ids`）
- `chats/cp_170000000.json`：单个串讲的对话历史
- 方法命名统一加 `cross_paper_` 前缀（如 `list_cross_paper_sessions`、`save_cross_paper_chat_history`）
- `add_papers_to_cross_paper_session(session_id, paper_ids)` 支持向已有串讲添加新论文
