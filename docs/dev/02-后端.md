## 配置管理 (`config.py`)

使用 `pydantic-settings` 管理配置，支持环境变量（前缀 `IPAPER_`）和配置文件。

```python
class Settings:
    data_dir: Path = ~/.ipaper         # 数据根目录
    host: str = "127.0.0.1"
    port: int = 3000
    llm: LLMConfig                     # LLM 子配置

class LLMConfig:
    api_base: str = "https://openrouter.ai/api/v1"
    api_key: str = ""                  # 需要用户配置（OpenRouter API Key）
    model: str = "google/gemini-3.1-pro-preview"
    temperature: float = 0.7
    max_tokens: int = 32768  # Gemini 3.1 Pro 的 reasoning tokens 也计入 max_tokens，留足空间避免正文被截断

class ProfileAnalysisConfig:
    model: str = "anthropic/claude-opus-4.6"  # 共用 llm 的 api_base 和 api_key
    temperature: float = 0.2
    max_tokens: int = 16384
```

- 配置文件：`~/.ipaper/config.json`
- 全局单例：`settings = Settings()`
- 启动时自动创建 `~/.ipaper/` 和 `~/.ipaper/papers/` 目录
- 调用 `settings.save_config()` 持久化到文件

| `CrossPaperAddPapersRequest` | 向串讲添加论文请求 (`paper_ids[]`) |
| `CrossPaperChatRequest` | 串讲对话请求 (`message`, `quotes?`, `page_selections?`) |
| `CrossPaperChatHistory` | 串讲对话历史 (`session_id`, `paper_ids[]`, `messages[]`, `forks?`) |

### 对话 (`/api/chat/{paper_id}/{session_id}`)

| 方法 | 路径 | 说明 | 请求体 | 响应 |
|------|------|------|--------|------|
| POST | `/api/chat/{id}/{sid}` | 发送消息 | `{ message, quotes?: [{text, source}], page_selections?: [{ paper_id?, ranges: [{start, end}] }] }` | SSE 流；若需选页则返回 `409` |
| GET | `/api/chat/{id}/{sid}/history` | 对话历史 | - | `ChatHistory`（含 `forks`、`draft`） |
| PUT | `/api/chat/{id}/{sid}/history` | 更新历史 | `{ messages, forks? }` | `{ message }` |
| PUT | `/api/chat/{id}/{sid}/history/draft` | 更新未发送草稿 | `{ draft: { input, quotes?, page_selections? } }` | `{ message }` |
| DELETE | `/api/chat/{id}/{sid}/history` | 清空历史 | - | `{ message }` |

**SSE 流格式：**

每行格式为 `data: {JSON}\n\n`，JSON 结构：

```json
// 正常响应片段
{ "type": "chunk", "content": "一段文字" }

// 响应因达到 max_tokens 被截断（在 done 之前发出，assistant 消息会标 truncated=true 落盘）
{ "type": "truncated", "reason": "length" }

// 响应完成（无论是否截断都会收到；finish_reason 可能是 "stop"/"length"/null）
{ "type": "done", "full_response": "完整响应文本", "finish_reason": "stop" }

// 错误（按异常类型返回中文消息）
{ "type": "error", "message": "API Key 无效或已过期，请在设置中检查" }
// 其他错误消息：API 额度不足或请求过于频繁 / 无法连接到 LLM 服务 / LLM 服务返回错误 (status_code)
```

当 PDF 已经转成图像但 payload 仍超过 20MB 时，接口不会再默认截取前几页，而是返回 `409`：

```json
{
  "detail": {
    "code": "page_selection_required",
    "message": "《某论文》转成图像后仍超过 20MB，请先选择要保留的页码范围。",
    "requirements": [
      {
        "paper_id": "2511.18870",
        "title": "HunyuanVideo 1.5 Technical Report",
        "total_pages": 42,
        "selected_ranges": [{ "start": 1, "end": 5 }]
      }
    ]
  }
}
```

## 服务层详解

### ArxivService (`services/arxiv_service.py`)

负责论文的下载和本地管理。

**arXiv ID 解析 (`parse_arxiv_input`)：**

支持以下输入格式：
- `2301.12345`
- `2301.12345v1`
- `https://arxiv.org/abs/2301.12345`
- `https://arxiv.org/pdf/2301.12345.pdf`

**下载流程 (`download_paper`) — 异步两阶段：**
1. 解析 arXiv ID
2. 若 `~/.ipaper/data/<user_id>/papers/{id}/meta.json` 已存在：按 `download_status` 分支
   - `ready` → 直接返回已有 meta（去重）
   - `downloading` → 返回现有 meta，并确保后台 task 在跑（进程重启后重新挂起时使用）
   - `failed` → 重置为 `downloading` 重新触发后台下载
3. 新论文：`asyncio.to_thread` 调 `arxiv.Client().results()` 同步拿 metadata（title/summary/authors），写入占位 `meta.json`（`download_status="downloading"`），更新 `index.json`，**立即返回**
4. 与此同时 `asyncio.create_task(_download_pdf_async)` 启动后台任务：
   - `asyncio.to_thread(paper.download_pdf, ...)` 下载 PDF（原子写：`paper.pdf.part` → rename 到 `paper.pdf`）
   - 成功：更新 `meta.json` 为 `download_status="ready"` + 调 `sync_service.request_sync("paper-ready", ...)`
   - 失败：按指数退避重试（2s / 4s / 8s，最多 3 次），仍失败则标记 `download_status="failed"` + 写入 `download_error`
   - 被 cancel（如用户删除论文）直接退出，由 `sync_service.delete_paper` 调 `arxiv_service.cancel_download` 触发

**为什么这样设计：**
- `arxiv.Client().results()` 和 `paper.download_pdf()` 都是同步 blocking 调用，直接在 async 路由里会卡住整个事件循环。用 `to_thread` 丢到线程池后，其它 API（论文列表、对话、翻译状态等）不再受影响。
- 添加按钮的 loading 只等 arxiv metadata（~1-2s），英文 PDF 在后台下载，用户可继续添加下一篇。
- 中文翻译仍由 `translation_service.ensure_translation` 独立异步处理，二者互不阻塞。

**索引管理：**
- `index.json` 位于 `~/.ipaper/data/<user_id>/papers/index.json`
- 格式：`{ papers: [{ arxiv_id, title, download_time }] }`
- 添加/删除论文时自动维护；`downloading` 状态也会写进 `index.json`（保持列表顺序稳定）

**PaperMeta 字段：**
- `download_status: "downloading" | "ready" | "failed"`（默认 `ready`，旧 meta 自动兼容）
- `download_error: Optional[str]`（失败原因，供前端展示）

**注意事项：**
- arXiv API 有速率限制（HTTP 429），建议使用 `delay_seconds=3.0`, `num_retries=3`；当前 `arxiv.Client()` 未配置这些参数，但 `_download_pdf_async` 自身的 3 次指数退避可部分缓解
- **后端进程重启时的孤儿下载**：若进程崩溃时有论文停在 `downloading` 状态，重启后 `meta.json` 里这些论文会保持 `downloading`，但没有后台 task。下次用户对同一篇论文再调 `POST /papers`，`_ensure_download_task` 会自动挂起新 task 恢复下载；不主动触发则保持 `downloading` 状态不变。

### LLMService (`services/llm_service.py`)

负责与 LLM API 通信。

**API 调用方式：**
- 使用 `openai` Python SDK（AsyncOpenAI）
- API 提供商：OpenRouter（`https://openrouter.ai/api/v1`）
- 讲解模型：`google/gemini-3.1-pro-preview`
- Reasoning：通过 `extra_body={"reasoning": {"effort": "medium"}}` 启用 Gemini 思考模式

**PDF 上传方式：**

将 PDF 文件编码为 base64，作为 `file` 类型的消息内容发送（OpenRouter 格式）：

```python
{
    "type": "file",
    "file": {
        "filename": "paper.pdf",
        "fileData": "data:application/pdf;base64,{pdf_base64}"
    }
}
```

对于大 PDF：
- `PDF_SIZE_THRESHOLD = 15MB` 时不再发送原始 PDF，而是用 PyMuPDF 逐页渲染 JPEG
- `IMAGE_PAYLOAD_LIMIT = 20MB` 限制的是 JPEG base64 payload 总量
- 若未提供 `page_selections` 且超限，抛出 `PageSelectionRequiredError`
- 若提供的页码范围仍超限，也会再次抛出 `PageSelectionRequiredError`，并把之前的 `selected_ranges` 带回前端，要求继续缩小
- 串讲模式会先遍历所有论文，把所有需要选页的论文一次性收集后统一返回

**消息构建逻辑 (`_build_messages`)：**
1. 添加 system prompt（学术论文阅读助手角色）
2. 在**第一条用户消息**中附加 PDF（base64）
3. 如果有 `quotes`，在最后一条用户消息前拼接引用文本（带来源标注：`[来自论文]` / `[来自对话]`）
4. assistant 消息附加 `reasoning` 字段（如有），传回 API 保持推理连续性
5. 后续消息正常传递

**System Prompt 要点：**
- 使用中文回答，保留英文术语
- 基于论文内容回答，不编造
- 可引用具体章节和图表

**串讲模式 (`chat_stream_cross_paper`)：**
- 使用 `get_cross_paper_system_prompt()` — 在基础用户画像 prompt 后追加串讲专用指令（对比分析框架、`[[id]]` 引用标注格式）
- `_build_cross_paper_first_user_content()` — 在第一条 user 消息中注入所有论文 PDF（每篇前加 `=== 论文 [[id]]: 标题 ===` 标注），复用 `_build_pdf_content_blocks()` 处理大 PDF，并汇总所有需要用户选页的论文
- `_build_messages_cross_paper()` — 构建完整的 API 消息列表

**流式响应 (`chat_stream` / `chat_stream_cross_paper`)：**

返回 `AsyncGenerator[str, None]`，逐 chunk yield 文本。两个可选的「副产物」参数由调用方传入可变容器来收集信息：
- `reasoning_collector: List[str]` — 流式过程中累积 reasoning tokens 片段，调用方在结束后自行 `''.join(...)`
- `finish_reason_holder: dict` — 流结束后写入 `{"value": "stop" | "length" | ...}`，`chat.py` 路由据此判断是否被 `max_tokens` 截断；若为 `"length"` 则向 SSE 发送 `{"type":"truncated","reason":"length"}` 事件，并把 assistant 消息的 `truncated=True` 一起落盘。

### StorageService (`services/storage_service.py`)

管理多会话对话历史的持久化。

**存储位置：** `papers/{arxiv_id}/chats/`

```
chats/
├── sessions.json         # 会话索引
└── {session_id}.json     # 单个会话历史
```

**sessions.json 结构：**
```json
{
  "sessions": [{ "id": "s_170000000", "title": "自动讲解", "created_at": "...", "updated_at": "..." }],
  "last_active_session_id": "s_170000000"
}
```

**会话文件结构：**
```json
{
  "paper_id": "2301.12345",
  "session_id": "s_170000000",
  "messages": [
    { "role": "user", "content": "这篇论文讲了什么?", "quotes": [{ "text": "...", "source": "pdf" }] },
    { "role": "assistant", "content": "这篇论文提出了...", "reasoning": "（可选）" }
  ],
  "forks": { "1": { "alternatives": [[...], [...]], "active": 1 } },
  "draft": {
    "input": "还没发送的问题",
    "quotes": [{ "text": "...", "source": "chat" }],
    "page_selections": [{ "paper_id": "2301.12345", "ranges": [{ "start": 1, "end": 5 }] }]
  }
}
```

**Session 管理方法：**
- `list_sessions(paper_id)` → `SessionList`
- `create_session(paper_id, title?)` → `SessionMeta`
- `delete_session(paper_id, session_id)` → `bool`
- `set_last_active_session(paper_id, session_id)`

**对话历史方法：**
- `get_chat_history(paper_id, session_id)` → `(List[ChatMessage], forks_dict, draft_dict)`
- `save_chat_history(paper_id, session_id, messages, forks?, draft?)` → 覆盖写入
- `clear_chat_history(paper_id, session_id)` → 删除文件

**引用持久化：** 用户消息上的 `quotes` 会跟随 `ChatMessage` 一起写入历史文件；单论文和串讲接口都采用同一结构，前端刷新后可直接恢复引用块显示

**未发送草稿持久化：** 当前会话的输入框文本、待发送引用，以及已经确认过的 `page_selections` 都保存在同一个会话文件的 `draft` 字段中。前端输入过程中会调用 `/history/draft` 接口同步；切换会话和刷新后通过 `GET /history` 恢复；真正发送消息时会把 `draft` 清空，但会把本轮使用的 `page_selections` 一并写回，供同一会话后续继续复用

**向后兼容：** 首次访问论文时，旧 `chat_history.json` 自动懒迁移到 `chats/` 目录

**串讲存储：** 独立于单论文，位于 `~/.ipaper/cross-paper/`：
- `sessions.json`：串讲会话索引（每个 session 额外存 `paper_ids`）
- `chats/cp_170000000.json`：单个串讲的对话历史
- 方法命名统一加 `cross_paper_` 前缀（如 `list_cross_paper_sessions`、`save_cross_paper_chat_history`）
- `add_papers_to_cross_paper_session(session_id, paper_ids)` 支持向已有串讲添加新论文
