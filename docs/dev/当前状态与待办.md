## 已知问题 (Bugs)

### P0 - 影响核心体验

| 问题 | 描述 | 根因分析 |
|------|------|---------|
| 搜索跳转后不居中 | 搜索结果跳转后，匹配项位于页面顶部而非中间 | `scrollIntoView` 和手动滚动调整均因找不到正确的滚动容器而失效。@react-pdf-viewer 使用虚拟滚动，实际滚动容器可能不是 `.rpv-core__inner-pages` |

### P1 - 可用但体验不佳

| 问题 | 描述 |
|------|------|
| Electron DevTools 自动打开 | `main.js` 中 `openDevTools()` 在开发模式下自动打开，可能需要移除或设为手动 |
| arXiv 速率限制 | `arxiv_service.py` 中 `arxiv.Client()` 未配置 `delay_seconds` 和 `num_retries`，频繁添加论文可能失败 |
| 后端端口不一致 | `config.py` 中后端端口为 3000，但 `iPaper.app/Contents/MacOS/iPaper` 启动脚本中使用 8000，Vite 代理指向 3000。需统一 |
| `main.py` 路由冲突 | `GET /api/config` 同时在 `main.py` 和 `routers/config.py` 中定义 |
| 服务器内存受限 | AWS 服务器仅 472MB 内存，无法在服务器上执行 `npm run build`，需本地构建后上传 |

## 部署架构

```
用户浏览器（Mac Chrome / iPhone Safari）
        │
        ▼ HTTPS (port 443)
    ┌─────────┐
    │  Xray   │  ← TLS 终结，VPN 流量正常代理
    │         │     非 VPN 流量 fallback ↓
    └────┬────┘
         ▼ (port 8080, localhost)
    ┌─────────┐
    │  Nginx  │  ← /ipaper/ → 前端静态文件 (dist/)
    │         │     /ipaper/api/ → 反向代理到后端 (:3000)
    │         │     /disk/ → filebrowser
    │         │     /alist/ → alist
    │         │     全部带 Basic Auth
    └─────────┘
         ▼ (port 3000, localhost)
    ┌─────────┐
    │ FastAPI  │  ← systemd 管理 (ipaper-backend.service)
    │ Backend  │     venv 隔离 Python 依赖
    └─────────┘
```

### 部署操作速查

```bash
# 本地改代码后同步到服务器
git add -A && git commit -m "..." && git push
ssh aws "cd ~/iPaper && git pull"

# 如果改了前端代码，需要本地构建后上传
cd frontend && npm run build
cd .. && tar czf /tmp/dist.tar.gz -C frontend dist/
scp /tmp/dist.tar.gz aws:/tmp/
ssh aws "cd ~/iPaper/frontend && tar xzf /tmp/dist.tar.gz && rm /tmp/dist.tar.gz"

# 如果改了后端代码，重启后端
ssh aws "sudo systemctl restart ipaper-backend"

# 如果改了 nginx 配置
ssh aws "sudo /usr/sbin/nginx -t && sudo /usr/sbin/nginx -s reload"

# 查看后端日志
ssh aws "sudo journalctl -u ipaper-backend -f"
```

## 开发注意事项

### 前端修改须知

1. **@react-pdf-viewer 的虚拟滚动**：该库使用虚拟滚动渲染 PDF，很多常规的 CSS 布局技巧不适用。操作内部 DOM 时需通过 JS 查找实际的滚动容器
2. **插件实例化**：searchPlugin 等插件如果在组件每次渲染时重新创建，会导致搜索状态丢失。应在组件顶层创建或使用 useRef
3. **选中文字交互**：mouseup/mousedown 监听需限定在各自的容器 ref 内，否则 PDF 和 Chat 的引用按钮会互相干扰
4. **PDF Worker**：使用 CDN 地址 `https://cdnjs.cloudflare.com/ajax/libs/pdf.js/3.11.174/pdf.worker.min.js`，需确保版本与 pdfjs-dist 一致
5. **生产构建路径**：`vite.config.ts` 中 build 模式 `base: '/ipaper/'`，`api.ts` 中 API_BASE 基于 `import.meta.env.BASE_URL` 动态计算

### 后端修改须知

1. **LLM 客户端懒加载**：`LLMService.client` 属性检测 API Key 变化后会重建客户端
2. **PDF 上传方式**：通过 base64 编码整个 PDF 发送给 Gemini，大文件会产生较大请求体
3. **arXiv 速率限制**：频繁请求 arXiv API 会被限流（HTTP 429），建议添加延迟和重试
4. **SSE 流格式**：严格遵循 `data: {json}\n\n` 格式，前端按行解析
5. **画像分析使用独立模型**：`ProfileAnalysisConfig` 配置 Claude Opus 4.6，与讲解模型分离

### 服务器须知

1. **内存受限**：服务器仅 472MB，不能在服务器上执行 `npm run build`，必须本地构建
2. **Xray 不能动**：443 端口由 Xray 管理，Nginx 只监听 8080（localhost），改 Nginx 配置前必须 `nginx -t` 测试
3. **Python 虚拟环境**：后端依赖在 `/home/admin/iPaper/backend/venv/` 中，不污染系统 Python
